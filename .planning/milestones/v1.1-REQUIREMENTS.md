**Defined:** 2026-05-04

## v1.1 Requirements (Shipped)

### Storage & Persistence (STOR)

- [x] **STOR-01**: Every copy operation (`db2db`, `db2file`, `file2db`) is automatically recorded with its full parameter set and outcome — ✅ Validated via `tests/test_history.py` and `tests/test_history_cli.py`
- [x] **STOR-02**: History records are stored in a lightweight JSON file alongside existing environment config — ✅ Validated via `elm/core/config.py` `get_history_file()`
- [x] **STOR-03**: Each history record has a unique auto-incrementing integer ID — ✅ Validated via `HistoryRecorder._next_id()`
- [x] **STOR-04**: History records capture: operation type, source, target, query/table, mode, batch_size, parallel, timestamps (start/end), status (success/failure), record count, and error message if failed — ✅ Validated via `HistoryRecord` dataclass (14 fields)
- [x] **STOR-05**: History storage is bounded (configurable max entries, default 100) with FIFO eviction — ✅ Validated via `HistoryRecorder._trim_records()`

### CLI Commands (CLIC)

- [x] **CLIC-01**: User can list previous copy operations via `elm copy list` showing: ID, timestamp, operation type, source → target, table, status — ✅ Validated via `tests/test_history_cli.py`
- [x] **CLIC-02**: User can re-run a previous copy operation via `elm copy re-run <id>` using all original parameters — ✅ Validated via CLI tests and UAT
- [x] **CLIC-03**: User can edit and re-run a previous copy operation via `elm copy edit <id>` with optional flags to override any original parameter — ✅ Validated via CLI tests
- [x] **CLIC-04**: `elm copy list` supports optional filters: `--status`, `--source`, `--target`, `--limit` — ✅ Validated via `tests/test_history_cli.py`

### GUI History Panel (GUIR)

- [x] **GUIR-01**: GUI displays a new "History" tab with a scrollable list of previous copy operations — ✅ Validated via UAT Test 1
- [x] **GUIR-02**: Each history entry shows: ID, timestamp, operation type, source → target, table, status (color-coded) — ✅ Validated via UAT Test 2
- [x] **GUIR-03**: Each history entry has a "Re-run" button that executes the operation again with original parameters — ✅ Validated via UAT Test 5 (resolved after gap fix)
- [x] **GUIR-04**: Each history entry has an "Edit & Re-run" button that pre-fills the Operations tab form with that entry's parameters — ✅ Validated via UAT Test 6
- [x] **GUIR-05**: History list automatically refreshes after a new copy operation completes — ✅ Validated via UAT Test 7

## Out of Scope (v1.1)

| Feature | Reason |
|---------|--------|
| Full undo/rollback | Too complex for v1.1; requires transaction log |
| History export to CSV/JSON | Nice-to-have, defer to v1.2 |
| Cross-machine sync | Requires server/cloud storage, out of scope |
| History entry delete button | User suggestion, out of scope for v1.1 |
